# FN² — 分形负反馈节点智能体框架

（完整中文版）

我们相信，自治 Agent 的真正突破不是靠更大的模型、更多的工具、或者更聪明的 prompt。  
而是靠更扎实的**控制架构**——能够自我批判、递归拆解、显式管理不确定性、在边界时优雅求助或升级的系统。

FN² 是朝着这个方向的一个极简、早期的尝试：  
从一开始就围绕**负反馈闭环 + 分形递归**来设计，目标是为长期迭代、深度扩展、甚至未来的多 Agent 协作打下一个可靠的骨架。

## 我们的长期设计哲学

1. **推理的未来在端侧**  
   大部分推理最终都会跑到设备本地。小模型的能力限制的是处理复杂问题的上限，而不是做一个日常可靠助手的下限。现在的本地小模型其实已经足够支撑类似 OpenCLAW 风格的稳定 Agent。

2. **值得信赖的下限比不断刷新的上限更重要**  
   LLM 越来越强，天花板被不断抬高。但绝大多数真实场景里，用户真正想要的是一个可靠的下限：行为可预测、可验证、不突然失控。我们坚信，只有结构化的反馈控制闭环，才能真正做到这一点。

3. **成本、主权、安全与监管无法回避**  
   推理成本虽然在快速下降，但大规模、常驻式部署时，成本依然是核心约束。更不用说数据主权、安全性、政治与监管挑战。本地/端侧部署的潜力被充分挖掘之前，智能体的大规模落地时代还远未到来。**毕竟，LLM 是在所有人类知识的集体智慧上训练出来的 —— LLM首先应该被所有人直接访问，无论他来自哪里、他是谁。**

4. **人类组织才是持久的基底——智能体必须演化去适配它**  
   LLM 不会让人类组织消失。相反，LLM 和 Agent 会自动演化成匹配、强化人类组织（团队、层级、责任流）的形态，而人类组织本身也会随之简化、优化。这是康威定律在社会技术系统上的直接推论。  
   所以 Agent 架构必须**以人类为中心**来设计：升级/求助是第一等公民，权限边界被尊重，可审计性是内置的，与人类决策环路的衔接是基础，而不是补丁。

## 当前状态（2026 年 3 月）

- 核心状态机 + 事件总线（Board）已经稳定  
- 负反馈闭环（Analyzer → Executor → Synthesizer → Controller）已实现  
- dryrun/mock 模式对验证控制流非常友好  
- 真实 LLM 模式已能跑，但还很脆弱（prompt 工程 + 工具链还在早期）  
- 工具/技能调用链只有骨架（真实工具即将接入）

## 坦诚的局限

- 真实 LLM 模式不稳定（经常过度问问题、分解浅、子任务汇总弱）  
- 还没有长期记忆 / 跨任务学习  
- 工具生态很初级（搜索、代码执行、文件操作是下一个重点）  
- 没有生产级的安全、限流、持久化  
- 目前不追求 benchmark 排名，重点在架构本身

## 我们现在不是在做什么

- 不是又一个 ReAct / AutoGPT / LangGraph 的变种  
- 不是在秀 prompt 工程  
- 不是生产就绪的框架  
- 不是要立刻冲榜的系统

我们在试图搭建一个**控制骨架**，让以后真正强大的 Agent 能站在上面继续建。

## 我们想找的伙伴

如果你对下面这些点有强烈共鸣，我们特别希望能一起把这个方向做扎实：

- 对当前“看起来聪明、实际很脆”的 prompt-chaining 风格感到失望  
- 相信负反馈、自我反思、显式验证是 Agent 长期可靠的关键  
- 对递归分解 + 子任务智能汇总有真实兴趣和想法  
- 愿意花时间调 prompt、设计工具、思考 memory，而不是只想快速出 demo  
- 对多 Agent 协作、能力边界处理、长期演化有长期思考  
- 不怕从底层重构、愿意写测试、愿意写文档

## 如何参与

**快速上手贡献**  
- 优化 Analyzer / Matcher / Synthesizer 的 prompt（当前最大瓶颈）  
- 实现首批真实工具（web 搜索、python 执行、文件读写优先）  
- 为 Board 状态转换补充单元测试  
- 帮忙可视化任务树（CLI 或简单 web 界面）

**深度贡献方向**  
- 子任务结果的智能聚合与合成逻辑  
- 基于不确定性的重试/升级策略  
- 长期记忆模式（向量？键值？情节式？）  
- 多 Agent 协作原语  
- 反思/批判环路的独立分离

有架构想法请用 “Proposal:” 前缀开 issue。
